# 📆 Project Management for Product Manager

**Commercial Value:**
Ensure alignment across stakeholders to avoid wasted effort and late surprises.

**When to use / when not:**
- **When:** Building a new unit within a company or launching a larger product.
- **When not:** Iterating on existing features, or building new features that can ship within ~6 weeks.

**Problem:**
- Without project management: Launching large, complex projects without clear ownership, alignment, and deadlines will likely derail the project, waste effort on unimportant topics, and slow final delivery.
- With too much project management: Planning everything upfront in detail leads to endless meetings, shifting priorities, delayed delivery, and missed market opportunities.

> Project management has a bad reputation in tech because it is associated with waterfall methods and huge planning overhead. For complex projects, we need deadlines and alignment across stakeholders. Using core principles of project management achieves this without the downsides.

---

## Principles

1. **Thinking backwards:**
   - Start from the launch and work backwards on what is needed.
   - Ensures alignment when various units or companies are involved.

2. **Business Goals / Milestones:**
   - Set 2–3 milestones tied to business outcomes. If multiple companies or units are involved, each will have their own milestones.

3. **Ownership:**
   - Every deliverable has a single directly responsible person to create accountability.

4. **Deadlines:**
   - Deadlines are commitments, not suggestions. If a deadline is at risk, communicate early so the team can find a solution together.
   - Deadlines are tied to business goals/milestones, not to every individual task.

5. **Operating model:**
   - Break milestones into subtasks and verify whether the milestone can be delivered on time. In high-seniority teams, a few high-level subtasks may suffice. Less senior teams need more granularity.
   - Check-in cadence: roughly once per week. Increase to daily as you near the finish line or when things are not running smoothly.

6. **Transparency / Reporting:**
   - The project plan is your reporting — no extra summaries needed.
   - Label each subtask: **Todo / In Progress / Done / Delayed**.

---

## Definitions

- **Milestone** → A meaningful outcome that demonstrates progress (e.g., "Beta launch to 10 customers").

---

## Examples

1. **Thinking backwards**
   - Bad: Teams start planning from "what features can we build first" without considering the final launch.
   - Good: Start from the launch date → define what the customer-facing milestone looks like → work backwards.

2. **Business Goals / Milestones**
   - Bad: A project has 10+ milestones, half of which are technical subtasks. Business impact is unclear.
   - Good: Only 2–3 milestones tied to business outcomes (e.g., "customer can send a fixed forecast to the procurement team").

3. **Ownership**
   - Bad: "Customer onboarding flow" is shared across 3 people with no clear responsibility.
   - Good: "Customer onboarding flow" has a single owner. Others support, but the owner is accountable.

4. **Deadlines**
   - Bad: "We'll aim to release in October" — everyone assumes different timelines, scope keeps creeping.
   - Good: "First release goes live October 15." Risks are flagged early so scope can be adjusted.

5. **Operating Model**
   - Bad: "Customer Portal Live" milestone written down with no subtasks. Team members interpret it differently.
   - Good: Milestone broken into subtasks (Design approved, Backend ready, Frontend done, Testing complete). Weekly check-ins; daily stand-ups in the final sprint.

6. **Transparency / Reporting**
   - Bad: Project lead sends separate weekly PowerPoints. Tasks live in one system, reporting in another.
   - Good: The project plan is the reporting. Every subtask is labeled. Anyone can check live status.
